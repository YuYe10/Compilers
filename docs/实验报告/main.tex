\begin{abstract}
本报告详细介绍了 Mandrill v2.0 语言编译器的完整实现过程。编译器采用 Java 语言开发，使用 Gradle 作为构建工具。词法分析器和语法分析器均采用手写递归下降实现，语义分析器与编译器后端借助 ANTLR 工具生成。编译器输出文本汇编指令，由课程组提供的模拟器执行。

实现过程分为四个阶段：实验一实现词法分析器，能够正确识别关键字、标识符、常量和运算符；实验二实现语法分析器，构建抽象语法树；实验三实现语义分析器，进行类型检查和作用域分析；实验四实现代码生成器，生成虚拟机汇编指令。所有合法测试用例和语义错误测试用例均已通过。
\end{abstract}

\section{词法分析器}

\textbf{【本章写作提示】}请按课程要求，描述你手写实现的词法分析器。以下给出建议的写作结构与参考内容，请根据你的实际实现进行补充或调整。

\subsection{整体架构与字符扫描}

词法分析器采用 \texttt{PushbackReader} 作为核心 IO 组件，支持字符回退功能，便于处理多字符运算符的向前看操作。

字符扫描的关键技术细节包括：
\begin{itemize}
    \item \textbf{行号与列号维护}：使用两个整型变量 \texttt{line} 和 \texttt{column} 分别记录当前位置。遇到换行符 \texttt{'\textbackslash n'} 时，\texttt{line++} 且 \texttt{column = 1}；遇到回车符 \texttt{'\textbackslash r'} 时仅重置 \texttt{column = 1}。
    \item \textbf{Token 起始位置记录}：在扫描每个 Token 前，先保存当前的 \texttt{line} 和 \texttt{column} 值，作为该 Token 的位置信息。
    \item \textbf{Windows 换行符兼容}：通过检测 \texttt{'\textbackslash r\textbackslash n'} 序列来正确处理 Windows 格式的换行符。
\end{itemize}

\subsection{多字符运算符与关键字识别}

单字符运算符（如 \texttt{+}、\texttt{-}、\texttt{*}、\texttt{/}）直接通过字符匹配识别。对于多字符运算符（如 \texttt{==}、\texttt{!=}、\texttt{<=}、\texttt{>=}），采用向前看（look-ahead）机制：

\begin{lstlisting}[language=Java]
if (currentChar == '=') {
    int next = reader.read();
    if (next == '=') {
        return new Token(TokenType.EQUALITY, "==");
    } else {
        reader.unread();
        return new Token(TokenType.ASSIGN, "=");
    }
}
\end{lstlisting}

关键字识别通过 \texttt{HashMap<String, TokenType>} 查找表实现，时间复杂度为 O(1)。关键字列表包括：\texttt{if}, \texttt{else}, \texttt{while}, \texttt{read}, \texttt{put}, \texttt{write}, \texttt{get}, \texttt{func}, \texttt{global}, \texttt{local}, \texttt{return}, \texttt{break}, \texttt{continue}。

\subsection{字符串与字符常量}

字符串常量与字符常量的解析逻辑如下：

\begin{itemize}
    \item \textbf{转义字符处理}：支持 \texttt{\textbackslash\textbackslash}、\texttt{\textbackslash n}、\texttt{\textbackslash '}、\texttt{\textbackslash "} 等转义序列。在扫描过程中遇到 \texttt{'\textbackslash '} 时，读取下一个字符并根据转义规则进行转换。
    \item \textbf{错误检测}：若遇到文件结束符（EOF）仍未找到闭合的引号，则报告"未闭合字符串/字符常量"错误，并记录当前行号和列号。
\end{itemize}

\subsection{Token 输出与错误处理}

Token 输出格式为：\texttt{[TokenType, value, line:column]}。例如：\texttt{[IDENTIFIER, "main", 1:1]}。

错误处理策略：
\begin{itemize}
    \item 遇到非法字符（如 \texttt{@}、\texttt{\#} 除外的特殊字符）时，输出错误信息并继续扫描。
    \item 未闭合的字符串或字符常量会触发错误报告，但不会导致程序崩溃。
\end{itemize}

\section{语法分析器}

\textbf{【本章写作提示】}请描述你手写实现的语法分析器。以下给出建议的写作结构与参考内容。

\subsection{递归下降 Parser 的设计}

递归下降分析器为每个非终结符编写了对应的方法：

\begin{itemize}
    \item \texttt{parseProgram()} - 解析整个程序
    \item \texttt{parseFunctionDef()} - 解析函数定义
    \item \texttt{parseStatement()} - 解析语句
    \item \texttt{parseExpression()} - 解析表达式（按优先级分层）
    \item \texttt{parseAssignStatement()} - 解析赋值语句
\end{itemize}

表达式优先级分层设计如下：

\begin{verbatim}
Expression       ::= EqualityExp
EqualityExp      ::= RelationalExp { ("==" | "!=") RelationalExp }
RelationalExp    ::= AdditiveExp { ("<" | "<=" | ">" | ">=") AdditiveExp }
AdditiveExp      ::= MultiplicativeExp { ("+" | "-") MultiplicativeExp }
MultiplicativeExp::= Primary { ("*" | "/" | "%") Primary }
Primary          ::= IntegerConstant | CharacterConstant | StringConstant
                   | Identifier ("[" Expression "]")?
                   | "(" Expression ")"
                   | FunctionCall
                   | Read | Get
\end{verbatim}

以 \texttt{a + b * c} 为例，解析过程如下：
1. \texttt{AdditiveExp} 调用 \texttt{MultiplicativeExp}
2. \texttt{MultiplicativeExp} 解析 \texttt{b * c}（优先结合乘法）
3. 再将结果与 \texttt{a} 相加

\subsection{特殊语法结构的处理}

\begin{itemize}
    \item \textbf{数组初始化语句}：通过向前看两个 Token 来区分。若标识符后紧跟 \texttt{[]}，则为数组初始化；否则为普通赋值或数组元素访问。
    \item \textbf{dangling-else 歧义}：采用贪心策略，让 \texttt{else} 匹配最近的未匹配 \texttt{if}。在实现中，\texttt{if} 语句的 then 分支和 else 分支都要求使用花括号。
    \item \textbf{空语句}：单独的 \texttt{;} 被解析为空语句，不产生任何 AST 节点。
\end{itemize}

\subsection{错误处理策略}

语法分析器的错误处理策略：
\begin{itemize}
    \item 先检查词法错误，确保 Token 流有效。
    \item 遇到非预期 Token 时，输出错误信息（包含行号和列号）并尝试跳过该 Token 继续解析。
    \item 实现了简单的 panic mode 错误恢复，遇到错误时跳过直到遇到分号或右花括号。
\end{itemize}

\section{语义分析}

\textbf{【本章写作提示】}请描述你基于 ANTLR 实现的语义分析器。以下给出建议的写作结构与参考内容。

\subsection{符号表设计}

符号表采用三层结构：

\begin{itemize}
    \item \textbf{全局变量}：名称、类型（整数/数组）、全局数据区编号
    \item \textbf{函数定义}：名称、参数列表、返回值类型、入口标签、局部变量表
    \item \textbf{局部变量}：名称、类型、栈帧偏移
\end{itemize}

采用两遍扫描策略：
1. 第一遍：收集所有函数签名和全局变量
2. 第二遍：检查函数体中的变量引用和类型兼容性

这样做的原因是 Mandrill 支持函数前向调用，函数定义可以出现在调用之后。

\subsection{类型推断规则}

Mandrill 具有隐式类型推断特性：

\begin{itemize}
    \item \texttt{a = expr}：根据 \texttt{expr} 的类型推断 \texttt{a} 的类型
    \item \texttt{a[] = expr}：\texttt{a} 被推断为数组类型
    \item \texttt{local a;}：未指定类型，需在使用前赋值
    \item \texttt{local a[];}：显式声明为数组类型
    \item 未声明直接使用的变量默认为全局整数类型
\end{itemize}

\subsection{类型检查}

类型检查器实现了以下检查规则：

\begin{itemize}
    \item 赋值左右类型兼容性（整数不能赋值给数组，反之亦然）
    \item 数组索引必须为整数类型
    \item \texttt{break}/\texttt{continue} 仅出现在循环体内
    \item \texttt{return} 返回值与函数声明的一致性
    \item 函数调用实参与形参的个数及类型匹配
\end{itemize}

错误报告携带行列号信息，格式为：\texttt{[行:列] 错误类型：详细信息}

\section{Mandrill 编译器}

\textbf{【本章写作提示】}请描述你基于 ANTLR 实现的编译器。以下给出建议的写作结构与参考内容。

\subsection{中间代码与代码生成策略}

编译器采用直接由 AST 生成汇编的策略，未使用中间表示（如三地址码）。代码生成器递归遍历 AST，为每个节点生成对应的虚拟机指令。

临时变量通过栈操作管理，不需要额外的临时变量分配策略。

\subsection{控制流翻译}

\begin{itemize}
    \item \textbf{while 循环}：
\begin{verbatim}
loop_start:
    eval cond
    jump_if_false loop_end
    ; 循环体
    jump loop_start
loop_end:
\end{verbatim}

    \item \textbf{if-else}：
\begin{verbatim}
    eval cond
    jump_if_false else_label
then:
    ; then 分支
    jump end_label
else_label:
    ; else 分支
end_label:
\end{verbatim}

    \item \textbf{break 与 continue}：通过维护标签栈实现。\texttt{break} 跳转到循环结束标签，\texttt{continue} 跳转到循环开始标签。
\end{itemize}

\subsection{函数调用的翻译}

函数调用的汇编生成过程：

\begin{itemize}
    \item 调用者按从左到右顺序压入参数
    \item \texttt{jal} 指令保存 SP 与 PC，跳转到函数入口
    \item 被调用者将参数从栈中取出并存入局部变量
    \item 返回值通过栈顶传递
    \item \texttt{ret} 指令恢复调用者上下文
    \item 无返回语句的函数默认返回 0
\end{itemize}

\subsection{数组与字符串的处理}

\begin{itemize}
    \item \textbf{数组元素访问}：地址计算公式为 $\text{Address} = \text{Base}(a) + i \times 4$
    \item \textbf{字符串常量}：通过 \texttt{malloc} 分配内存，逐字符写入（UTF-32 编码），末尾添加 0
    \item \textbf{数组赋值 vs 字符串赋值}：数组赋值直接写入地址；字符串赋值先分配内存再写入内容
\end{itemize}

    \label{tab:instructions}
    \begin{tabular}{l|l|l}
    \hline
    类别 & 指令 & 说明 \\ \hline
    数据存取 & \texttt{dstore x} & 将立即数 \texttt{x} 压入操作数栈 \\
     & \texttt{dload x} & 从全局变量区读取变量 \texttt{x} 入栈 \\

    \end{tabular}

\subsection{汇编模拟器执行}

实验四的完整执行流程：

\begin{itemize}
    \item 编译器从标准输入读取源程序，输出文本汇编到标准输出
    \item 评测机将汇编重定向到临时文件并由模拟器加载执行
    \item 模拟器逐行解析文本汇编，忽略以 \texttt{\#} 开头的注释
    \item 模拟器的运行时内存模型包括：操作数栈、全局变量区、局部变量区、调用栈帧、堆内存
    \item 理解模拟器的工作原理有助于正确生成汇编代码，特别是栈操作顺序和函数调用约定
\end{itemize}

\section{心得体会}

\textbf{【本章写作提示】}请结合你的实际实现过程，撰写真实的感悟与反思。以下方向供参考：

手写词法分析器时，转义字符处理和行号列号维护花费了较多时间。特别是处理 Windows 和 Unix 换行符的兼容性问题，需要仔细考虑各种边界情况。

从手写前端转向 ANTLR 实现语义分析与代码生成，感受到了工具带来的便利。ANTLR 自动生成语法分析器，减少了大量重复代码。但原理性的东西如类型推断规则、代码生成策略仍需手动处理。

编译器后端中，控制流跳转和函数调用约定是难点。通过手绘栈状态图和单步跟踪，逐步理清了思路。特别是理解 \texttt{eval} 指令的栈操作顺序，需要先压入跳转地址再压入条件值。

项目采用 Gradle 多模块管理，模块划分清晰，依赖管理方便。每个实验可以独立测试，便于定位问题。
