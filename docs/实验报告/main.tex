\begin{abstract}
本报告详细介绍了 Mandrill v2.0 语言编译器的完整实现过程。编译器采用 Java 语言开发，使用 Gradle 作为构建工具。词法分析器和语法分析器均采用手写递归下降实现，语义分析器与编译器后端借助 ANTLR 工具生成。编译器输出文本汇编指令，由课程组提供的模拟器执行。

实现过程分为四个阶段：实验一实现词法分析器，能够正确识别关键字、标识符、常量和运算符；实验二实现语法分析器，构建抽象语法树；实验三实现语义分析器，进行类型检查和作用域分析；实验四实现代码生成器，生成虚拟机汇编指令。所有合法测试用例和语义错误测试用例均已通过。
\end{abstract}

\section{词法分析器}

词法分析是编译过程的第一阶段，其任务是将源程序的字符流转换为有意义的词素（Token）序列。本节详细介绍手写实现的词法分析器的整体架构与核心算法。

\subsection{整体架构与字符扫描}

词法分析器采用 \texttt{PushbackReader} 作为核心 IO 组件，该类支持字符回退功能，便于处理多字符运算符的向前看（look-ahead）操作。整体采用状态机驱动的扫描策略，根据当前状态和输入字符决定下一个状态和输出。

字符扫描的关键技术细节包括：

\begin{itemize}
    \item \textbf{行号与列号维护}：使用两个整型变量 \texttt{line} 和 \texttt{column} 分别记录当前位置。遇到换行符 \texttt{'\textbackslash n'} 时，\texttt{line++} 且 \texttt{column = 1}；遇到回车符 \texttt{'\textbackslash r'} 时仅重置 \texttt{column = 1}。
    \item \textbf{Token 起始位置记录}：在扫描每个 Token 前，先保存当前的 \texttt{line} 和 \texttt{column} 值，作为该 Token 的位置信息，便于后续错误报告。
    \item \textbf{Windows 换行符兼容}：通过检测 \texttt{'\textbackslash rtextbackslash n'} 序列来正确处理 Windows 格式的换行符，避免重复计数。
\end{itemize}

词法分析器的核心工作流程如下：首先跳过空白字符（包括空格、制表符、换行符等），然后根据当前字符的类型进入不同的识别状态。若当前字符是字母或下划线，则进入标识符/关键字识别状态；若当前字符是数字，则进入整数常量识别状态；若当前字符是单引号或双引号，则分别进入字符常量或字符串常量识别状态；若当前字符是其他符号，则尝试识别运算符或分隔符。

\subsection{多字符运算符与关键字识别}

单字符运算符（如 \texttt{+}、\texttt{-}、\texttt{*}、\texttt{/}）直接通过字符匹配识别。对于多字符运算符（如 \texttt{==}、\texttt{!=}、\texttt{<=}、\texttt{>=}），采用向前看机制：

\begin{lstlisting}[language=Java]
if (currentChar == '=') {
    int next = reader.read();
    if (next == '=') {
        return new Token(TokenType.EQUALITY, "==");
    } else {
        reader.unread();
        return new Token(TokenType.ASSIGN, "=");
    }
}
\end{lstlisting}

关键字识别通过 \texttt{HashMap<String, TokenType>} 哈希查找表实现，将所有关键字预先存入表中。扫描标识符时，首先收集完整的标识符字符串，然后在哈希表中查找。若找到匹配项，则返回对应的关键字 Token；否则返回标识符 Token。该设计的时间复杂度为 O(1)，具有较高的查找效率。

Mandrill 语言的关键字列表包括：\texttt{if}, \texttt{else}, \texttt{while}, \texttt{read}, \texttt{put}, \texttt{write}, \texttt{get}, \texttt{func}, \texttt{global}, \texttt{local}, \texttt{return}, \texttt{break}, \texttt{continue}。

\subsection{字符串与字符常量}

字符串常量与字符常量的解析是词法分析器中最复杂的部分，需要处理转义序列和引号匹配。

\begin{itemize}
    \item \textbf{转义字符处理}：支持 \texttt{\textbackslash\textbackslash}（反斜杠）、\texttt{\textbackslash n}（换行）、\texttt{\textbackslash r}（回车）、\texttt{\textbackslash t}（制表）、\texttt{\textbackslash '}（单引号）、\texttt{\textbackslash "}（双引号）等转义序列。在扫描过程中遇到 \texttt{'\textbackslash'} 时，读取下一个字符并根据转义规则进行转换，将转义序列转换为对应的实际字符值。
    \item \textbf{错误检测与报告}：若遇到文件结束符（EOF）仍未找到闭合的引号，则报告"未闭合字符串/字符常量"错误，并记录当前行号和列号。错误信息格式为：\texttt{[line:column] Error: Unterminated string constant}。
    \item \textbf{字符串长度限制}：根据 Mandrill 语言规范，字符串常量长度不超过 255 个字符（UTF-32 编码后每个字符占 4 字节），超出限制时报告相应错误。
\end{itemize}

\subsection{Token 输出与错误处理}

词法分析器的 Token 输出格式为：\texttt{[TokenType, value, line:column]}。例如：\texttt{[IDENTIFIER, "main", 1:1]} 表示在第 1 行第 1 列识别到一个标识符 main。

错误处理策略采用"记录并继续"模式：
\begin{itemize}
    \item 遇到非法字符（如非 ASCII 范围内的控制字符）时，输出错误信息并继续扫描后续字符。
    \item 未闭合的字符串或字符常量会触发错误报告，但不会导致程序崩溃，后续继续正常处理。
    \item 所有错误信息统一收集，在词法分析结束后一次性报告，便于用户一次性了解所有问题。
\end{itemize}

\section{语法分析器}

语法分析是编译过程的核心阶段，其任务是根据语言的语法规则，将词法分析器生成的 Token 序列构建成抽象语法树（Abstract Syntax Tree, AST）\footnote{参考 Aho et al., Compilers: Principles, Techniques, and Tools, 2006}。本节介绍递归下降语法分析器的设计与实现。

\subsection{递归下降 Parser 的设计}

递归下降分析器为每个非终结符编写了对应的解析方法，通过函数调用实现自顶向下的语法分析。Mandrill 语言的 EBNF 文法采用递归下降方法实现，消除了左递归以避免无限递归问题。

递归下降分析器为每个非终结符编写了对应的方法：

\begin{itemize}
    \item \texttt{parseProgram()} - 解析整个程序，入口方法
    \item \texttt{parseFunctionDef()} - 解析函数定义
    \item \texttt{parseStatement()} - 解析语句
    \item \texttt{parseExpression()} - 解析表达式（按优先级分层）
    \item \texttt{parseAssignStatement()} - 解析赋值语句
    \item \texttt{parseConditionStatement()} - 解析 if-else 语句
    \item \texttt{parseLoopStatement()} - 解析 while 循环语句
\end{itemize}

表达式优先级分层设计是递归下降分析器的核心。每个优先级对应一个非终结符和解析方法，高优先级的方法被低优先级的方法调用。这种设计自然地消除了左递归问题，同时保证了运算符的优先级和结合性。

表达式优先级分层设计如下：

\begin{verbatim}
Expression       ::= EqualityExp
EqualityExp      ::= RelationalExp { ("==" | "!=") RelationalExp }
RelationalExp    ::= AdditiveExp { ("<" | "<=" | ">" | ">=") AdditiveExp }
AdditiveExp      ::= MultiplicativeExp { ("+" | "-") MultiplicativeExp }
MultiplicativeExp::= Primary { ("*" | "/" | "%") Primary }
Primary          ::= IntegerConstant | CharacterConstant | StringConstant
                   | Identifier ("[" Expression "]")?
                   | "(" Expression ")"
                   | FunctionCall
                   | Read | Get
\end{verbatim}

以 \texttt{a + b * c} 为例，解析过程如下：
\begin{enumerate}
    \item \texttt{parseExpression()} 调用 \texttt{parseEqualityExp()}，进而调用 \texttt{parseRelationalExp()}，最后调用 \texttt{parseAdditiveExp()}
    \item \texttt{parseAdditiveExp()} 首先调用 \texttt{parseMultiplicativeExp()} 解析左侧操作数
    \item \texttt{parseMultiplicativeExp()} 解析 \texttt{b * c}（乘法优先级高于加法）
    \item 加法运算符导致创建加法表达式节点，其左右子节点分别为 \texttt{a} 和 \texttt{b * c}
\end{enumerate}

\subsection{特殊语法结构的处理}

Mandrill 语言的某些语法结构需要特殊处理：

\begin{itemize}
    \item \textbf{数组初始化语句}：通过向前看两个 Token 来区分数组初始化与数组元素访问。若标识符后紧跟 \texttt{[]}，则为数组初始化语句，语法为 \texttt{Identifier "[]" "=" Expression}；否则为普通赋值或数组元素访问语句。
    \item \textbf{dangling-else 歧义}：采用贪心策略，让 \texttt{else} 匹配最近的未匹配 \texttt{if}。Mandrill 语言的语法强制要求 if 和 while 语句必须使用花括号，这从根本上消除了 dangling-else 问题。
    \item \textbf{空语句}：单独的 \texttt{;} 被解析为空语句，不产生任何 AST 节点。
    \item \textbf{函数参数列表}：函数调用时，实参个数必须与形参个数匹配。解析函数调用时，首先解析左括号，然后循环解析参数直到遇到右括号。
\end{itemize}

\subsection{错误处理策略}

语法分析器的错误处理采用同步恢复机制：

\begin{itemize}
    \item \textbf{panic mode 恢复}：遇到语法错误时，跳过输入 Token 直到遇到同步点（分号、右花括号、右括号等），然后继续解析。这种方法简单有效，能避免级联错误报告。
    \item \textbf{错误信息报告}：每个解析方法在检测到错误时，输出包含当前 Token 和期望 Token 的错误信息。错误信息格式为：\texttt{[line:column] Syntax error: expected X, got Y}。
    \item \textbf{部分 AST 生成}：即使发生语法错误，Parser 仍尝试生成部分的 AST，仅包含能够正确解析的部分，以便后续阶段能够继续处理。
\end{itemize}

\section{语义分析}

语义分析是编译过程中进行类型检查和作用域分析的关键阶段，其任务是确保程序的语义正确性。本节介绍基于 ANTLR 生成的语法分析器构建的语义分析器的设计与实现。

\subsection{符号表设计}

符号表是语义分析的核心数据结构，用于存储和检索程序中标识符的相关信息{}\footnote{参考 Appel, Modern Compiler Implementation in Java, 2007}。符号表采用分层结构，支持作用域嵌套：

\begin{itemize}
    \item \textbf{全局变量}：存储在全局作用域，包括名称、类型（整数/数组）、全局数据区编号。
    \item \textbf{函数定义}：存储函数的完整签名，包括名称、参数列表、返回值类型、入口标签、局部变量表。
    \item \textbf{局部变量}：存储在函数作用域，包括名称、类型、栈帧偏移。
\end{itemize}

采用两遍扫描策略进行语义分析：
\begin{enumerate}
    \item 第一遍：收集所有函数签名和全局变量，构建函数调用图
    \item 第二遍：检查函数体中的变量引用和类型兼容性
\end{enumerate}

这种设计的原因在于 Mandrill 支持函数前向调用，即函数定义可以出现在调用之后。第一遍扫描确保在检查函数体时，所有被调用的函数签名已经存在于符号表中。

符号表的实现采用哈希表和栈的结合。全局作用域使用哈希表存储所有全局变量和函数；每个函数作用域使用栈来管理局部变量，支持作用域的进入和退出操作。

\subsection{类型推断规则}

Mandrill 具有隐式类型推断特性，变量的类型根据其首次出现的形式推断：

\begin{itemize}
    \item \texttt{a = expr}：根据 \texttt{expr} 的类型推断 \texttt{a} 的类型。若右侧是整数表达式，则 \texttt{a} 为整数类型；若右侧是数组访问，则 \texttt{a} 为整数类型（因为数组元素是整数）。
    \item \texttt{a[] = expr}：\texttt{a} 被推断为数组类型。方括号明确标识数组声明。
    \item \texttt{local a;}：未指定初始类型，需在使用前通过赋值语句推断类型。
    \item \texttt{local a[];}：显式声明为数组类型，无需类型推断。
    \item 未声明直接使用的变量默认为全局整数类型，这是 Mandrill 语言的默认规则。
\end{itemize}

类型推断在语义分析阶段完成，推断结果存储在符号表中，供后续的类型检查使用。

\subsection{类型检查}

类型检查器实现了以下检查规则，确保程序的类型安全：

\begin{enumerate}
    \item \textbf{赋值兼容性检查}：赋值语句的左右两边类型必须兼容。整数不能直接赋值给数组变量，数组变量也不能直接赋值给整数变量。
    \item \textbf{数组索引检查}：数组索引表达式必须为整数类型。使用非整数类型作为索引会触发类型错误。
    \item \textbf{控制流语句检查}：\texttt{break}/\texttt{continue} 语句只能出现在循环体内部。在非循环位置使用这些语句会报告语义错误。
    \item \textbf{返回语句检查}：\texttt{return} 语句只能出现在函数内部。在全局作用域使用 return 语句会报告语义错误。
    \item \textbf{函数调用检查}：函数调用的实参个数和类型必须与函数声明的形参匹配。参数个数或类型不匹配会报告错误。
    \item \textbf{操作数类型检查}：算术运算符（\texttt{+}、\texttt{-}、\texttt{*}、\texttt{/}、\texttt{\%}）的操作数必须为整数类型。
    \item \textbf{条件表达式检查}：while 循环条件和 if 语句条件必须为整数类型（零为假，非零为真）。
\end{enumerate}

错误报告携带行列号信息，格式为：\texttt{[line:column] Semantic error: [错误类型] [详细信息]}。

\section{Mandrill 编译器}

编译器后端负责将经过语义检查的抽象语法树转换为目标代码。本节介绍代码生成器的设计与实现，包括中间表示、代码生成策略和目标代码格式。

\subsection{中间代码与代码生成策略}

编译器采用直接由 AST 生成汇编的策略，未使用中间表示（如三地址码）。这种设计简化了编译器结构，降低了实现复杂度。代码生成器递归遍历 AST，为每个节点生成对应的虚拟机指令。

代码生成的基本策略是后序遍历 AST：首先为子节点生成代码，然后将子节点的计算结果组合为当前节点的结果。这种策略确保了子表达式的值在父表达式使用之前已经计算完成并位于栈顶。

临时变量通过栈操作管理，不需要额外的临时变量分配策略。虚拟机的栈结构天然支持临时值的存储，每个计算步骤的结果都位于栈顶，便于后续指令使用。

\subsection{控制流翻译}

控制流语句的翻译需要生成跳转指令，实现程序流的分支和循环。

\begin{itemize}
    \item \textbf{while 循环}：生成循环结构需要维护循环入口标签和循环出口标签。
\begin{verbatim}
loop_start:
    ; 生成条件表达式代码，结果在栈顶
    eval cond
    jump_if_false loop_end
    ; 生成循环体代码
    jump loop_start
loop_end:
\end{verbatim}

    \item \textbf{if-else 语句}：条件分支需要维护条件为真和为假两个标签。
\begin{verbatim}
    ; 生成条件表达式代码，结果在栈顶
    eval cond
    jump_if_false else_label
then:
    ; 生成 then 分支代码
    jump end_label
else_label:
    ; 生成 else 分支代码
end_label:
\end{verbatim}

    \item \textbf{break 与 continue}：通过维护标签栈实现。当进入循环时，将当前循环的入口标签和出口标签压入标签栈。\texttt{break} 指令从栈顶弹出出口标签并生成跳转指令；\texttt{continue} 指令从栈顶弹出入口标签并生成跳转指令。这种设计支持嵌套循环中 \texttt{break}/\texttt{continue} 语句的正确行为。
\end{itemize}

\subsection{函数调用的翻译}

函数调用是编译器后端最复杂的部分，涉及参数传递、上下文保存和恢复等操作。

函数调用的汇编生成过程如下：

\begin{enumerate}
    \item 调用者按从左到右顺序压入参数（使用 \texttt{dstore} 指令）
    \item \texttt{jal} 指令保存当前执行上下文（SP 和 PC），跳转到函数入口
    \item 被调用函数开始执行，首先将参数从栈中取出并存入局部变量区
    \item 执行函数体，生成相应的汇编代码
    \item \texttt{ret} 指令恢复调用者上下文，返回到调用点的下一条指令
    \item 函数返回值通过栈顶传递，调用者使用栈顶的值作为函数调用的结果
\end{enumerate}

无返回语句的函数默认返回 0。在函数入口处，代码生成器自动插入一条 \texttt{dstore 0} 指令，确保函数总是有返回值。

\subsection{数组与字符串的处理}

\begin{itemize}
    \item \textbf{数组元素访问}：地址计算公式为 $\text{Address} = \text{Base}(a) + i \times 4$，其中 Base(a) 是数组的基地址，i 是索引，4 是每个元素的字节大小。
    \item \textbf{字符串常量}：通过 \texttt{malloc} 指令分配内存，逐字符写入（UTF-32 编码），末尾添加 0 作为字符串结束标志。
    \item \textbf{数组赋值}：直接将源数组的地址复制到目标变量，不进行元素级别的复制。
    \item \textbf{字符串赋值}：先释放目标字符串原有内存，再分配新内存并复制字符串内容。
\end{itemize}

\subsection{汇编模拟器执行}

实验四的完整执行流程如下：

\begin{enumerate}
    \item 编译器从标准输入读取 Mandrill 源程序，进行词法分析、语法分析和语义分析
    \item 代码生成器遍历 AST，生成文本格式的虚拟机汇编代码
    \item 汇编代码输出到标准输出，评测机将其重定向到临时文件
    \item 模拟器加载临时文件中的汇编代码，逐行解析并执行
    \item 模拟器维护运行时内存模型：操作数栈、全局变量区、局部变量区、调用栈帧、堆内存
    \item 程序执行完成后，模拟器输出程序结果
\end{enumerate}

模拟器的设计使得编译器开发者可以专注于代码生成逻辑，而无需实现完整的运行时环境。理解模拟器的工作原理对于正确生成汇编代码至关重要，特别是栈操作顺序和函数调用约定。

\section{心得体会}

通过本次编译原理课程实践，我对编译器设计与实现有了更深入的理解，以下是几点主要体会。

手写词法分析器时，转义字符处理和行号列号维护花费了较多时间。特别是处理 Windows 和 Unix 换行符的兼容性问题，需要仔细考虑各种边界情况。手写实现虽然代码量较大，但让我对词法分析的原理有了更深刻的理解。相比之下，使用 Lex 等自动化工具虽然能快速生成代码，但可能掩盖了底层实现的细节。

从手写前端转向 ANTLR 实现语义分析与代码生成，我感受到了工具带来的便利。ANTLR 自动生成语法分析器，减少了大量重复代码。但类型推断规则、作用域分析、代码生成策略等核心逻辑仍需手动实现，这些才是编译原理课程的精髓所在。

编译器后端中，控制流跳转和函数调用约定是最大的难点。\texttt{eval} 指令的栈操作顺序特别容易出错：条件跳转时需要先压入 ELSE 地址，再压入 THEN 地址，最后压入条件值。我通过手绘栈状态图和单步跟踪，逐步理清了思路，最终正确实现了代码生成。

测试驱动开发在本次实践中发挥了重要作用。每个阶段完成后，我都会运行大量的测试用例来验证实现的正确性，包括合法用例和错误用例。这种方法帮助我在早期发现并修复了大量 bug，避免了问题的累积和蔓延。

\section{总结}

本项目成功实现了 Mandrill v2.0 语言的完整编译器，包括词法分析器、语法分析器、语义分析器和代码生成器四个核心模块。

词法分析器采用状态机驱动的扫描策略，支持关键字、标识符、常量和运算符的识别，具有完善的错误检测和报告机制。语法分析器采用递归下降方法实现，通过优先级分层设计自然地处理了运算符的优先级和结合性。语义分析器基于 ANTLR 生成的语法分析器构建，实现了符号表管理和全面的类型检查。代码生成器将抽象语法树转换为虚拟机汇编指令，支持控制流、函数调用、数组和字符串等核心特性。

所有测试用例均已通过，包括合法用例（覆盖表达式计算、条件语句、循环语句、函数调用、数组操作等场景）和语义错误用例（覆盖类型错误、作用域错误、控制流错误等场景）。通过本次实践，我深入理解了编译器的完整工作流程，掌握了词法分析、语法分析、语义分析和代码生成的核心技术，达到了编译原理课程的教学目标。
